To **automatically clear or bypass the cache** in your app when you deploy a new version (especially to fix the `Unexpected token '<'` issue from stale JS), there are a few techniques you can code into your **Next.js** or frontend app.

---

## ✅ Best Practices to Auto-Clear Cache

### 1. **Enable Versioned Asset Hashing (built-in to Next.js)**

Next.js already does this by default — every build generates unique file names like:

```
_app-a1b2c3d4.js
```

This ensures that **new builds don't reuse old cached JS files** — unless you have custom headers or misconfigured service workers.

➡️ Just make sure you're using `next build` and not custom export shims.

---

### 2. **Force Clients to Fetch Latest Assets (Meta or JS)**

You can add cache-busting logic like this:

#### Option A: Meta tag to prevent caching

In your `pages/_document.js`:

```js
import { Html, Head, Main, NextScript } from 'next/document'

export default function Document() {
  return (
    <Html>
      <Head>
        <meta httpEquiv="Cache-Control" content="no-store, no-cache, must-revalidate, max-age=0" />
        <meta httpEquiv="Pragma" content="no-cache" />
        <meta httpEquiv="Expires" content="0" />
      </Head>
      <body>
        <Main />
        <NextScript />
      </body>
    </Html>
  )
}
```

> 💡 Downside: This disables all caching, which can affect performance. Use only if your app changes frequently or you want users to *always* load the latest.

---

### 3. **Automatically Unregister Old Service Workers**

If you're using a service worker (e.g. via PWA or custom logic), make sure it unregisters on update:

```js
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.getRegistrations().then(function (registrations) {
    for (let registration of registrations) {
      registration.unregister();
    }
  });
}
```

Place this inside a client-side file, like a layout effect or `useEffect()` in `_app.js`.

---

### 4. **Version Check with LocalStorage or API (Recommended for SPAs)**

In `_app.js`, check for version mismatch:

```js
useEffect(() => {
  const CURRENT_VERSION = '1.0.3' // bump this on deploy
  const storedVersion = localStorage.getItem('app_version')

  if (storedVersion && storedVersion !== CURRENT_VERSION) {
    localStorage.setItem('app_version', CURRENT_VERSION)
    window.location.reload(true) // full reload
  } else {
    localStorage.setItem('app_version', CURRENT_VERSION)
  }
}, [])
```

> 🔁 This ensures that when you deploy `v1.0.4`, everyone gets a clean reload.

---

### 5. **Set HTTP Headers on Vercel (if you're using custom headers)**

In `next.config.js`:

```js
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
        ],
      },
    ]
  },
}
```

---

## 🚀 Recommendation

For most Next.js projects, use:

* **Built-in file hashing**
* **Service worker unregistration**
* **Version-based cache busting (via localStorage or API ping)**

This combo gives performance *and* avoids stale file issues.
